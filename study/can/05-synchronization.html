CH5. 동기화

학습 목표

  • CAN이 별도 클럭선 없이 동기를 유지하는 두 축(Hard Sync, Resynchronization)의 차이를 설명한다.
  • Phase Error의 부호에 따라 Phase_Seg1 / Phase_Seg2 중 어느 쪽을 조정하는지 도출한다.
  • ISO 11898-1 기반으로 허용 가능한 오실레이터 편차를 계산한다.
  • 장거리·고속·CAN FD 환경별로 sample point 배치 전략을 고른다.

CAN은 물리적으로 TX/RX 전용선과 공통 GND 말고는 없다. 클럭선이 없는 asynchronous 버스다. 그런데도 수십 개 노드가 1 Mbps로 충돌 없이 중재하고 같은 비트를 동시에 읽는다. 비결은 두 단계의 동기화 메커니즘, Hard Synchronization과 Resynchronization이다. 이번 챕터는 그 두 메커니즘이 CH4에서 설계한 Bit Time 위에서 어떻게 작동하는지 본다.

동기화 문제는 임베디드 현장에서 가장 디버깅이 어려운 영역 중 하나다. 하드웨어 파형은 정상처럼 보이고, 컨트롤러 설정값도 수치상으로는 맞아떨어지는데 유독 특정 노드만 통신이 끊기는 경우가 대부분 이 카테고리에 속한다. 원인을 한 번 정확히 짚어내려면, CAN 컨트롤러가 비트 경계에서 어떤 판단을 하는지, 그리고 어떤 조건에서 재동기화가 실패하는지를 수식 수준까지 따라가야 한다. 이 챕터의 목표는 그런 판단 기준을 숫자로 바꿔두는 것이다.

1. 왜 동기화가 필수인가

각 노드는 자체 크리스털에서 클럭을 만든다. 두 크리스털은 제조 편차·온도 변화로 미세하게 다른 주파수를 가진다. 편차가 0.1%뿐이어도, 1000 비트를 연속으로 전송하면 송신자와 수신자의 시간 차이는 한 bit 넘게 벌어진다. CAN 프레임은 최대 100bit 이상이고 연속 bit 사이에도 edge가 한참 없을 수 있기 때문에, 노드들은 기회가 올 때마다 edge를 보고 자기 시간을 재맞춰야 한다.

또한 편차가 작더라도 주변 환경(온도, 전압, 진동)에 따라 크리스털의 주파수는 시시각각 변한다. 차량 환경은 영하 40도에서 영상 85도까지 사용 범위가 넓고, 농기계는 진동이 추가로 더해진다. 그래서 단순히 스펙상의 ppm 값만 믿어서는 안 되고, 최악의 운용 조건까지 고려해서 동기화 여유를 잡아야 한다.

이 "시간을 재맞추는 기회"가 두 가지다.

  1. 프레임 시작 순간 — recessive에서 dominant로 바뀌는 SOF의 edge로 강제로 리셋.
  2. 프레임 진행 중 아무 edge — recessive→dominant 전환이 생길 때마다 Phase 세그먼트를 미세 조정.

2. Hard Synchronization

Hard Sync는 프레임의 시작점에서만 발생한다. 버스가 idle(recessive)일 때 어떤 노드가 SOF(0, dominant)를 드라이브하면 그 순간의 recessive→dominant edge를 모든 수신 노드가 감지한다. 수신 노드는 이 edge를 자신의 Sync_Seg 끝에 맞춰 bit time을 완전히 재시작한다. 이 시점 이후부터 같은 time base로 비트를 세기 시작한다.

특징:

  • 프레임 시작에서 단 한 번 발생.
  • 기존에 진행 중이던 bit time counter를 무시하고 리셋.
  • SJW 제약을 받지 않는다(대신 프레임 중에는 쓰이지 않는다).

Hard Sync가 필요한 이유는 버스 idle 상태가 얼마나 길어졌는지 예측할 수 없기 때문이다. 프레임이 전혀 없는 상태에서 수 초 또는 수 분이 지난 뒤에 새 프레임이 시작되면, 각 노드의 누적 drift가 허용 범위를 한참 넘었을 수 있다. 이럴 때 단순 resync로는 회복이 불가능하므로, 아예 SOF에서 bit time을 완전히 다시 세는 방식이 필요하다.

3. Resynchronization

프레임이 진행되는 동안 노드의 크리스털 편차로 time base가 어긋나기 시작한다. 이때 dominant로 전환되는 모든 edge에서 Resynchronization이 일어난다. Hard Sync와 달리, Bit Time을 완전히 리셋하지 않고 Phase 세그먼트만 ±SJW 범위 내에서 조정한다.

Phase Error

Phase Error e는 edge가 예상 Sync_Seg 대비 얼마나 어긋났는지다.

  • e > 0: edge가 늦게 도착(송신자가 수신자보다 느림).
  • e < 0: edge가 일찍 도착(송신자가 수신자보다 빠름).
  • e = 0: edge가 Sync_Seg 안에 정확히 들어옴.
sequenceDiagram
    participant Bus as Bus Edge
    participant Node as Receiving Node
    participant BTL as Bit Time Logic
    Note over Node: Bit Time 시작 (Sync_Seg)
    Bus->>Node: edge 감지 (recessive→dominant)
    Node->>BTL: Phase Error e 계산
    alt e > 0 (edge 늦음)
        BTL->>BTL: Phase_Seg1 += min(e, SJW)
    else e < 0 (edge 이름)
        BTL->>BTL: Phase_Seg2 -= min(|e|, SJW)
    else e = 0
        BTL->>BTL: 조정 없음
    end
    Note over BTL: Sample Point 유지
    BTL->>Node: Sample (dominant/recessive)

조정 규칙

  • e > 0 (늦음): Phase_Seg1을 min(e, SJW) TQ만큼 늘린다. Sample Point도 그만큼 뒤로 밀려 edge와의 정렬이 맞춰진다.
  • e < 0 (빠름): Phase_Seg2를 min(|e|, SJW) TQ만큼 줄인다. 다음 bit time이 더 일찍 시작돼 빠른 송신자를 따라간다.
  • 동일 비트 안에서는 한 번만 조정. 추가 edge는 무시.
  • 송신 중인 노드에선 자기 자신이 만든 edge에 대해서는 resync하지 않는다(자기 기준이 맞으므로).

여기서 중요한 점은 resync가 Phase 세그먼트만 건드린다는 것이다. Sync_Seg와 Prop_Seg는 변하지 않는다. Prop_Seg를 건드리지 않는 이유는, 이 구간이 물리적으로 신호가 전파되는 데 필요한 시간이기 때문에 "시계를 더 돌리는" 것으로는 해결되지 않는 영역이기 때문이다. Phase 세그먼트는 순수한 버퍼 영역이라, 편차 보정용으로 늘리거나 줄여도 비트 샘플링의 정확도에 미치는 영향이 적다.

4. 비트 시간 내부 상태 기계

동기화 동작을 하드웨어 관점에서 그려보면, 한 비트를 처리하는 동안 컨트롤러는 여러 개의 서브 상태를 순차적으로 거친다. 이 상태 전이는 상태 머신으로 표현할 수 있으며, edge가 들어올 때마다 Phase_Seg1 또는 Phase_Seg2의 길이가 동적으로 조정된다. 디지털 로직 레벨에서는 TQ 카운터를 매 클럭마다 증가시키다가, 상태 전이 시점에 Phase_Seg 카운터를 늘리거나 줄이는 방식으로 구현된다.

stateDiagram-v2
    [*] --> Sync_Seg
    Sync_Seg --> Prop_Seg: +1 TQ
    Prop_Seg --> Phase_Seg1: Prop_TQ 소진
    Phase_Seg1 --> SamplePoint: Phase1_TQ 소진
    SamplePoint --> Phase_Seg2: 버스값 읽기
    Phase_Seg2 --> [*]: Phase2_TQ 소진 → 다음 bit
    Phase_Seg1 --> Phase_Seg1: Edge 감지 시 +SJW(Resync)
    Phase_Seg2 --> [*]: Edge 감지 시 -SJW(조기 종료)

Sample Point는 Phase_Seg1이 끝나는 순간 1회 찍힌다. CAN은 single sampling이 기본이지만, 컨트롤러에 따라 3회 다수결 sampling 옵션(MCP2515의 CNF2.SAM)도 있다. 3회 샘플링은 노이즈가 심한 환경에서 순간적인 글리치를 걸러내는 데 도움이 되지만, 샘플링에 필요한 시간이 길어져 고속(1 Mbps 이상)에서는 쓰기 어렵다. 대부분의 차량용 트랜시버는 내부적으로 잡음 필터링을 수행하므로, 단일 샘플링으로도 충분히 안정적이다.

5. 오실레이터 편차 허용도

이제 본격적인 수식이 등장한다. 두 노드가 서로 얼마나 다른 주파수로 동작해도 여전히 통신이 가능한지를 결정하는 것은, 결국 Phase_Seg1과 Phase_Seg2 그리고 SJW의 상대적 크기다. 이 편차 허용도를 계산하는 것은 새로운 MCU를 도입하거나 오실레이터 부품을 변경할 때마다 반드시 거쳐야 하는 단계다.

두 노드의 클럭 주파수 차이를 df(fractional, 예: 0.005 = 0.5%)라 할 때, ISO 11898-1은 허용 조건을 다음 두 식으로 묶어준다(간소화):

df ≤ min(Phase_Seg1, Phase_Seg2) / (2 × (13 × Bit_TQ − Phase_Seg2))
df ≤ SJW / (20 × Bit_TQ)

둘 중 더 엄격한 쪽이 실제 허용치를 결정한다.

예시 계산

Bit_TQ=16, Prop=5, P1=7, P2=3, SJW=3일 때:

  • 첫 식: min(7,3) / (2×(13×16 − 3)) = 3 / (2×205) = 0.00732 → 0.73%.
  • 둘째 식: 3 / (20×16) = 0.00937 → 0.94%.
  • 엄격한 쪽은 0.73%. 즉 약 ±0.36%의 개별 노드 편차까지 허용.

Bit_TQ=20, Prop=7, P1=7, P2=5, SJW=4:

  • 첫 식: 5 / (2×(13×20 − 5)) = 5/510 = 0.98% → 0.98%.
  • 둘째 식: 4 / (20×20) = 1.0% → 1.0%.
  • 허용 ~0.98%, 개별 노드 ±0.49%.

숫자가 말하는 것

크리스털 오실레이터는 보통 ±30~100 ppm(0.003~0.01%) 수준이라 여유가 크다. 세라믹 오실레이터는 ±0.5%를 넘기기도 해서 위 식에서 margin이 바닥난다. CAN 노드가 크리스털을 권장받는 이유가 여기 있다.

위 식의 의미를 조금 더 풀어보면 다음과 같다. 첫 번째 식은 "한 프레임 안에서 누적되는 drift"를 기준으로 한 제약이고, 두 번째 식은 "재동기화가 따라갈 수 있는 최대 속도"를 나타낸다. 둘 중 작은 값이 최종 허용 편차를 결정한다는 것은, 어느 한쪽이라도 부족하면 통신이 실패한다는 뜻이다. Phase_Seg2가 너무 작으면 첫 번째 식이 타이트해지고, SJW가 작으면 두 번째 식이 병목이 된다. Bit Time 파라미터를 설계할 때 두 값을 균형 있게 배분해야 하는 이유다.

실제로 bit timing을 고를 때 이 편차 계산을 먼저 돌려보면, 동일한 baud rate에서도 어떤 조합은 1% 편차까지 버티고 어떤 조합은 0.3%도 못 버틴다는 것을 알 수 있다. 양산 단계에서 크리스털의 ppm 스펙이 바뀌거나 벤더 변경이 필요할 때, 이 수식이 바로 허용 여부를 판단하는 기준이 된다.

6. Bit Stuffing이 만드는 edge 기회

동기화는 edge가 있어야만 동작한다. 그런데 CAN 프레임의 데이터에는 0이 연속으로 나오거나 1이 연속으로 나오는 구간이 얼마든지 있을 수 있다. 만약 10비트 연속 dominant가 흘러간다면, 수신 노드는 그 시간 동안 단 한 번도 edge를 보지 못하고 자체 클럭으로만 비트를 세야 한다. 이 경우 drift가 누적되어 특정 시점부터 샘플링이 틀어진다. CAN이 이 문제를 해결하기 위해 도입한 장치가 bit stuffing이다.

연속된 같은 레벨의 비트가 5개를 넘지 못하도록 CAN은 stuff bit를 자동으로 삽입한다(송신자가 5bit 같은 값을 보낸 직후 반대 레벨 1bit 삽입, 수신자가 제거). Stuffing의 부수효과는 최대 5 bit마다 edge를 강제로 만든다는 것이다. 이 edge 하나하나가 resynchronization의 기회가 된다. 클럭 drift가 누적되기 전에 재동기할 수 있어, 긴 프레임에서도 sample point가 유지된다.

CAN FD는 payload가 최대 64바이트까지 커져 프레임이 훨씬 길지만, stuffing 규칙을 유지하고(4-bit stuff count 필드를 별도로 추가) 동일한 방식으로 edge 기회를 보장한다. 다시 말해, CAN은 stuffing이라는 간단한 규칙만으로 "반드시 일정 간격마다 edge가 생긴다"는 수학적 보장을 얻어낸다. 이 보장이 깨지면 동기화 체계 자체가 흔들리므로, 트랜시버가 stuff error를 감지했을 때 프레임을 재전송하는 동작은 단순한 에러 처리가 아니라 동기화 유지의 핵심이다.

7. Sample Point 배치 전략

Sample Point 위치는 단순한 설정 값이 아니라, 네트워크 토폴로지와 운용 환경에 직결된 설계 결정이다. 아래 표는 현업에서 많이 쓰이는 조건-SP 쌍을 정리한 것이다. 표를 그대로 따르기보다는, 각 상황이 왜 그런 SP를 요구하는지를 이해하고 본인의 환경에 맞춰 응용하는 것이 좋다.

8. CAN FD와 Partial Bit Timing

CAN FD에서 동기화가 더 어려워지는 이유는 프레임 안에서 비트 속도가 두 번 바뀌기 때문이다. Arbitration 구간에선 Classical CAN과 같은 느린 속도로 동기화를 유지하다가, BRS 비트를 지나면 훨씬 빠른 속도로 전환된다. 이 전환 지점에서 다시 시간 기준을 잡고, 데이터 구간 내에서도 자체적인 resync를 수행해야 한다.

CAN FD는 프레임 내에서 arbitration 구간과 data 구간의 bit rate가 다르다. 그래서 BTR도 두 세트다.

  • Nominal Bit Timing(NBTP): arbitration · control · CRC · EOF 구간용. Classical CAN과 같은 느린 rate(예: 500 kbps).
  • Data Bit Timing(DBTP): BRS 비트 이후 data payload 구간용. 빠른 rate(예: 2 Mbps).
  • Transmitter Delay Compensation(TDC): data 구간에서 자신이 송신한 edge를 자기 자신이 올바르게 다시 샘플링하도록 보상. DBTP와 함께 설정한다.

ISO CAN FD는 non-ISO 초기안에 stuff-count CRC와 TDC 세부를 추가한 표준이다. 두 변형은 서로 호환되지 않으므로 컨트롤러·툴의 ISO/Non-ISO 플래그를 반드시 맞춘다. 최근 양산되는 모든 신형 차량과 농기계는 ISO CAN FD를 쓰기 때문에, 특별한 이유가 없다면 ISO 모드를 기본으로 둔다. 반면 오래된 연구용 개발 보드나 초기 구현체는 non-ISO를 쓰는 경우가 있어, 혼용 환경에서는 프레임이 이유 없이 거부되는 현상을 종종 본다.

9. 실전 계측: Sample Point 확인

이론적인 계산이 끝났다면, 마지막은 실측이다. 오실로스코프 또는 로직 분석기를 이용한 계측은 bit timing의 정확성을 확인하는 가장 확실한 방법이다. 특히 양산 이전 단계에서는 여러 보드를 계측해 편차 분포를 확인하는 것이 품질 확보의 핵심이다.

실제 파형에서 sample point를 확인하는 절차:

  1. 오실로스코프로 CANH/CANL 차동 파형과 MCU의 RX 핀을 동시에 캡처한다.
  2. SOF edge를 trigger로 잡는다.
  3. 각 bit 경계에서 RX 핀 level이 컨트롤러 내부 bit 결과로 반영된 타이밍을 본다. Bit Time 시작점 기준 SP 비율 계산.
  4. 계산 SP(설계값)와 실측 SP가 ±2 TQ 이내면 정상. 큰 차이가 나면 BRP·TSx 설정 또는 클럭 소스 확인.
  5. CAN FD는 arbitration/data 구간 각각 측정. Data 구간 SP가 80% 미만이면 TDC 재설정.

계측 없이 계산만으로 bit timing을 확정하는 것은 위험하다. 실제 하드웨어에는 트랜시버의 개별 편차, PCB 임피던스, 접지 노이즈, 커넥터 저항 같은 변수가 개입하므로, 오실로스코프로 실측한 SP와 계산값 사이에 1~2 TQ 정도의 차이는 흔히 나타난다. 최종 양산 단계에서는 여러 대의 샘플 보드를 계측해 편차 범위를 확인하고, 필요하면 Phase_Seg 배분을 다시 조정한다.

Logic Analyzer 접근

Saleae/Kingst 같은 로직 분석기는 디지털 edge만 보지만, 대부분의 CAN 디코더가 sample point를 시각화하는 마커 기능을 제공한다. 개발 보드 RX 라인만 연결해도 비트 타이밍 오류 중 80%는 이 도구로 잡힌다.

이외에도 로직 분석기의 디코더 기능을 쓰면 bit timing 에러와 동기화 상태를 함께 볼 수 있다. 예를 들어 Saleae Logic Pro는 각 비트에 대해 "Sync", "Sample", "Phase" 시점을 마커로 표시해주며, 에러 프레임이 발생하면 어느 비트에서 오류가 생겼는지 즉시 파악할 수 있다. 이런 도구를 활용하면 단순 파형 해석만으로는 보이지 않던 타이밍 불일치를 쉽게 찾을 수 있다.

자주 하는 실수

  • CAN FD data 구간 sample point를 NBTP와 같은 비율로 두면 빠른 rate에서 bit error가 쏟아진다. DBTP 별도 설정 필수.
  • TDC를 빼먹으면 2 Mbps 이상에서 자기 자신의 dominant를 recessive로 잘못 읽는다.
  • 세라믹 오실레이터는 온도 변화 시 drift가 커 resync로도 못 따라잡는다. 차량·농기계 환경은 크리스털 고정.
  • Stub이 길면 반사파가 edge jitter로 잡혀 Phase Error가 크게 보인다. 원인은 타이밍이 아니라 토폴로지.

다음 챕터

다음 CH6. Classical CAN 심화에서는 이 bit 위에 쌓이는 프레임 구조로 시야를 올린다. Stuff/CRC 계산, Remote/Overload/Error 프레임, 중재의 경계 상황까지 파고들며 동기화가 흔들릴 때 어떤 에러 프레임이 나오는지 연결해 본다.

핵심 정리

  • CAN은 클럭선이 없고, edge 기반 동기화로 같은 비트를 같은 순간에 샘플링한다.
  • Hard Sync는 SOF에서 1회, bit time을 완전히 리셋한다.
  • Resync는 프레임 중 edge마다 일어나며, Phase Error 부호에 따라 Phase_Seg1 연장 / Phase_Seg2 단축(최대 SJW).
  • 오실레이터 편차 허용도는 Phase_Seg 크기와 SJW로 결정되며, 크리스털(~0.01%)은 여유롭고 세라믹(~0.5%)은 부족.
  • CAN FD는 NBTP/DBTP 분리와 TDC로 data 구간 sample point를 따로 관리한다.
