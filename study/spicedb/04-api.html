CH4. API 전체

학습 목표

  • SpiceDB의 5개 핵심 API가 각각 어떤 질문에 답하는지 한 장으로 잡는다.
  • Check / Expand / LookupResources / LookupSubjects / Watch의 내부 동작과 비용을 구분한다.
  • 각 API에 맞는 실전 사용처(문서 열람, 공유 UI, 검색 필터, 캐시 동기화)를 매칭한다.
  • API별 흔한 실수 패턴을 미리 알아둔다.

API 지도

SpiceDB API는 많아 보이지만, 결국 "누가 무엇에 대해 권한을 가졌는가"를 다른 각도에서 묻는 것이다. 다섯 개를 질문 형태로 정리하면 아래처럼 된다.

이제 각각을 뜯어보자.

Check API

Check는 SpiceDB에서 가장 많이 호출되는 hot path다. 모든 요청마다 호출될 수도 있는 API라, latency와 throughput 최적화가 전부 여기에 집중돼 있다.

요청은 단순하다.

resource:   document:readme
permission: view
subject:    user:alice

응답은 세 가지 중 하나다.

  • PERMISSIONSHIP_HAS_PERMISSION — 있다.
  • PERMISSIONSHIP_NO_PERMISSION — 없다.
  • PERMISSIONSHIP_CONDITIONAL_PERMISSION — Caveat이 걸려 있는데 context가 부족해서 판정 유보.

세 번째 값은 Caveat 때문에 등장한다. 예를 들어 valid_ip(ip) caveat이 걸린 relation인데 요청 context에 ip가 안 실려 왔다면, 서버는 "조건을 평가할 수 없다"는 뜻으로 conditional을 반환한다. 클라이언트는 어떤 context가 부족한지 보고 다시 시도할 수 있다.

Go 코드로 보면 아래와 같다.

resp, err := client.CheckPermission(ctx, &pb.CheckPermissionRequest{
    Resource:   &pb.ObjectReference{ObjectType: "document", ObjectId: "readme"},
    Permission: "view",
    Subject: &pb.SubjectReference{
        Object: &pb.ObjectReference{ObjectType: "user", ObjectId: "alice"},
    },
    Consistency: &pb.Consistency{
        Requirement: &pb.Consistency_AtLeastAsFresh{AtLeastAsFresh: zedToken},
    },
})
if resp.Permissionship == pb.CheckPermissionResponse_PERMISSIONSHIP_HAS_PERMISSION {
    // 통과
}

Python이라면 이렇다.

resp = client.CheckPermission(CheckPermissionRequest(
    resource=ObjectReference(object_type="document", object_id="readme"),
    permission="view",
    subject=SubjectReference(
        object=ObjectReference(object_type="user", object_id="alice"),
    ),
    consistency=Consistency(at_least_as_fresh=zed_token),
))
allowed = resp.permissionship == CheckPermissionResponse.PERMISSIONSHIP_HAS_PERMISSION

CheckBulkPermissions

같은 subject에 대해 여러 resource를 한 번에 확인하고 싶을 때가 있다. 페이지에 문서 100개를 띄우는데 각 문서별로 view/edit 버튼 노출 여부를 판단하는 경우 같은 상황이다. 100번의 Check를 RPC 100번으로 보내는 대신 CheckBulkPermissions에 100개의 (resource, permission, subject) 쌍을 묶어 보낼 수 있다.

서버 측은 내부적으로 최대한 공통 부분을 캐시/재활용해 개별 Check보다 훨씬 적은 비용으로 처리한다.

CheckBulkPermissions vs LookupResources

이 둘은 혼동하기 쉽다. CheckBulkPermissions는 resource 목록이 이미 결정된 상황에서 각각 권한을 확인한다. 반면 LookupResources는 resource 목록 자체를 권한으로부터 역산한다. DB 쿼리 결과가 먼저 있고 그 위에 권한 필터를 얹는 UX라면 CheckBulkPermissions가 맞고, 처음부터 "이 user가 볼 수 있는 모든 문서"를 구하고 싶다면 LookupResources다.

Expand API

Expand는 Check와 짝을 이루는 API다. 입력은 (resource, permission) 두 개이고, 출력은 "이 permission을 가진 멤버 userset의 트리"다. Check가 boolean 하나만 반환하는 것과 달리, Expand는 평가 트리를 통째로 반환한다.

트리가 뭐가 들어있는지 보면 감이 온다.

Expand(document:readme, view)
  union
  ├── _this: [user:alice, user:bob]
  ├── computed_userset(editor)
  │     └── _this: [user:carol]
  └── tuple_to_userset(parent -> view)
        └── folder:docs (더 파고들 수 있는 하위 트리)

쓰임새는 명확하다. 권한 관리 / 공유 다이얼로그 / 감사 화면처럼 "멤버를 시각적으로 보여줘야 하는" UI에 쓴다. 같은 화면을 Check로 구현하려면 후보 user 전체에 대해 N번 Check를 날려야 하는데, Expand 한 번이면 트리를 받아 화면에서 펴 주면 된다.

다만 비용이 싸지 않다.

  • 트리의 깊이와 너비가 커질수록 응답 크기도 커진다.
  • hot path(API 가드)에 박으면 지연과 대역폭이 모두 문제 된다.
  • 권한 판단용이 아니라 UI 렌더용이라는 전제로 쓴다.

Zanzibar 논문에서도 이 API는 고의적으로 중간 트리를 "펴지 않고" 반환한다. computed_userset과 tuple_to_userset은 가지로 남겨 두고, 필요하면 클라이언트가 더 파고들게 한다. 그래야 대규모 그룹에서도 응답이 폭발하지 않는다.

LookupResources

역방향 질의다. "Alice가 view 권한을 가진 모든 document의 id는?"

stream, _ := client.LookupResources(ctx, &pb.LookupResourcesRequest{
    ResourceObjectType: "document",
    Permission:         "view",
    Subject: &pb.SubjectReference{
        Object: &pb.ObjectReference{ObjectType: "user", ObjectId: "alice"},
    },
    OptionalCursor: cursor, // 페이지네이션
    OptionalLimit:  100,
})
for {
    resp, err := stream.Recv()
    if err == io.EOF { break }
    ids = append(ids, resp.ResourceObjectId)
}

대표적인 UX는 세 가지다.

  • 내 문서 목록, 내가 속한 프로젝트 목록 같은 "my stuff" 화면.
  • 검색 UI에서 DB 검색 결과를 권한으로 post-filter.
  • 외부 시스템(Elastic, Postgres)에 권한 기반 필터 조건을 넘겨야 할 때 id 목록.

cursor 기반 페이지네이션이 지원된다. 첫 호출로 cursor를 받고, 다음 호출에 그 cursor를 끼워 이어 받는다.

성능 주의점

LookupResources는 subject의 권한 범위가 커질수록 느려진다. 한 user가 수만 개 resource에 권한을 가진 상황이면 답을 만드는 데 시간이 걸린다. 실전에서는 대개 이런 패턴으로 감싼다.

  • 검색어 기반 후보 resource를 DB에서 먼저 좁힌다.
  • 그 id 목록에 대해 CheckBulkPermissions로 권한 확인.
  • 페이지가 끝나면 다음 페이지로.

"전체 resource에서 권한 있는 것만 필터"를 LookupResources 하나로 해결하려 들면 대규모 사용자에게서 금세 문제 된다는 걸 기억해 두자.

LookupSubjects

반대 방향이다. "document:readme의 view 권한을 가진 모든 user는?"

stream, _ := client.LookupSubjects(ctx, &pb.LookupSubjectsRequest{
    Resource:          &pb.ObjectReference{ObjectType: "document", ObjectId: "readme"},
    Permission:        "view",
    SubjectObjectType: "user",
})

용도는 "공유 대상 후보" UI에 가장 많다. 공유 화면을 열었을 때 현재 접근 권한이 있는 사용자 목록을 보여주거나, 반대로 "아직 권한이 없는 사람"을 계산할 때 쓴다.

LookupResources와 LookupSubjects는 이름이 비슷하지만 사용 빈도가 꽤 다르다. 일반적으로는 LookupResources가 훨씬 많이 쓰인다. 대부분의 UI는 "이 user가 볼 수 있는 것들"이 관심사이지, "이 resource를 볼 수 있는 사람들"이 아니기 때문이다. LookupSubjects는 공유/관리 UI에서만 간헐적으로 돌아간다.

Watch API

권한 상태의 변경 스트림이다. relationship이 추가/삭제되면 이벤트로 흘러나온다. gRPC 서버 스트리밍으로 열어 두고 받아 쓰는 형태다.

주요 용도는 다음과 같다.

  • 애플리케이션 측 캐시 무효화 — 권한이 바뀌면 해당 resource 관련 캐시를 날린다.
  • 외부 시스템 동기화 — Elasticsearch의 _acl 필드, Redis의 권한 인덱스, BigQuery의 감사 테이블에 반영.
  • 감사 로그 — 누가 누구에게 언제 권한을 줬는지의 실시간 로그.
stream, _ := client.Watch(ctx, &pb.WatchRequest{
    OptionalStartCursor: lastCursor, // 마지막으로 처리한 지점부터 재개
})
for {
    resp, err := stream.Recv()
    if err != nil { /* 재연결 */ break }
    for _, upd := range resp.Updates {
        applyToCache(upd)
    }
    lastCursor = resp.ChangesThrough
}

Watch는 cursor 기반이다. 수신 중에 연결이 끊기면 마지막 cursor부터 다시 이어 받을 수 있다. 실전 패턴(재연결, 뒤떨어짐 감지, 멱등 적용)은 CH12. 리소스·관계 동기화에서 본격적으로 다룬다.

Watch cursor는 반드시 durable하게 저장하자

Watch 소비자는 잦은 재시작을 가정해야 한다. 메모리에만 cursor를 두면 프로세스 재기동 시 "언제부터 놓친 이벤트가 있는지" 알 수 없게 된다. 최소한 로컬 파일이나 datastore에 주기적으로 커밋해 두고, 재시작 시 그 지점부터 Watch를 연다.

실전 시나리오: 검색 UI

검색창에서 "문서 제목으로 검색 후 권한으로 필터"를 구현하는 전형적인 흐름이다.

sequenceDiagram
    participant U as 사용자
    participant App as App Server
    participant ES as Elasticsearch
    participant S as SpiceDB

    U->>App: "budget"으로 문서 검색
    App->>ES: 제목 match 쿼리
    ES-->>App: doc:1, doc:2, ..., doc:50
    App->>S: CheckBulkPermissions<br>(user:alice, view, doc:1..50)
    S-->>App: doc:1, doc:3, doc:7 ... 권한 있음
    App-->>U: 권한 있는 문서만 렌더

여기서 LookupResources를 먼저 부르지 않은 이유는 단순하다. 제목 필터가 강하므로 DB가 50개로 먼저 좁혀주고, 그 위에 BulkCheck를 한 번 돌리는 게 훨씬 싸다. 만약 "사용자가 볼 수 있는 전체 문서" 같은 전체 목록 페이지라면 LookupResources가 맞다. 상황에 따라 두 접근을 섞는다.

흔한 실수

피해야 할 패턴

  • Expand를 매 요청마다 호출 — 지연과 대역폭이 금방 터진다. UI 렌더 한정으로만 쓴다.
  • LookupResources 결과를 캐시하지 않음 — 동일한 "my stuff" 화면을 짧은 시간 내 여러 번 계산하게 된다.
  • Watch 재연결 시 cursor를 초기화 — 누락이 발생한다. cursor는 반드시 durable storage에.
  • CheckPermission을 loop로 N번 — CheckBulkPermissions으로 묶자.
  • Check의 consistency를 항상 fully_consistent — hot path라면 대부분 at_least_as_fresh로 충분하다. CH3. ZedToken의 옵션 표를 다시 보자.

핵심 정리

핵심 정리

  • Check는 hot path. boolean 하나를 빠르게. CheckBulkPermissions으로 묶을 수 있으면 묶는다.
  • Expand는 UI 렌더용. 멤버 트리를 통째로 받아 화면에서 그린다. hot path 금지.
  • LookupResources는 "내가 볼 수 있는 것들" 역방향 질의. 권한 범위가 크면 DB 후보를 먼저 좁히고 BulkCheck로 필터하는 패턴이 안전하다.
  • LookupSubjects는 "이걸 볼 수 있는 사람들". 공유/관리 UI에서 주로 쓴다.
  • Watch는 변경 이벤트 스트림. 캐시 무효화·외부 동기화·감사 로그의 출발점이다. cursor를 durable storage에 저장하는 것이 실전의 핵심.
  • 성능 특성이 각각 다르다. 한 API에 모든 요구를 태우지 말고, 질문 형태별로 나눠 쓰자.
