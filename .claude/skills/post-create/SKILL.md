# post-create — 블로그 포스트 핑퐁 작성

사용자가 주제/개요를 주면 초안을 작성하고, 핑퐁 대화로 다듬어 `docs/_posts/{category}/`에 저장한다.

## 프로젝트 정보

- **프로젝트 루트**: `/Users/hobeen/private/hobeen-kim.github.io`
- **포스트 디렉토리**: `docs/_posts/{category}/`
- **이미지 디렉토리**: `docs/.vuepress/public/images/{slug}/`
- **카테고리**: architecture, conference, culture, database, infra, java, kafka, spring, tech

## 워크플로우

### 1단계: 주제 파악

사용자가 주제를 던지면, 아래를 확인한다:

- **주제**: 무엇에 대해 쓸 것인가
- **카테고리**: 어느 카테고리에 넣을 것인가 (불확실하면 제안)
- **배경**: 왜 이 글을 쓰려 하는가 (실무 경험? 학습 정리? 삽질 기록?)

사용자가 개요까지 줬다면 바로 2단계로 넘어간다. 주제만 줬다면 개요를 먼저 제안한다.

### 2단계: 개요(목차) 잡기

포스트의 전체 구조를 목차 형태로 제안한다:

```
## 1. 도입 — 왜 이 주제를 다루는가
## 2. 핵심 개념 — ...
## 3. 실제 적용 — ...
## 4. 정리
```

사용자가 목차를 조정하면 반영한다. 목차가 확정되면 3단계로 넘어간다.

### 3단계: 초안 작성

확정된 목차를 기반으로 전체 초안을 작성한다.

#### 글쓰기 원칙

이 블로그의 포스트는 개발자가 실무에서 겪은 경험과 기술적 깊이를 담는 글이다. 단순한 공식문서 번역이 아니라 "내가 왜 이걸 알아봤고, 어떻게 이해했는가"가 드러나야 한다.

- **문체**: 해라체 (`~이다`, `~한다`, `~있다`). 경어체 금지.
- **도입부**: 이 글을 쓰게 된 실무적 맥락이나 동기로 시작한다. "회사에서 ~를 하면서", "~를 적용하다가 궁금해서" 같은 자연스러운 시작.
- **설명 방식**: 개념을 나열하지 말고, 왜 이게 필요한지 → 어떻게 동작하는지 → 실제로 어떻게 쓰는지 순서로 풀어간다.
- **1인칭 사용 자연스럽게**: "나도 처음엔 헷갈렸다", "이 부분이 핵심이다" 같은 주관적 표현을 적절히 섞는다.
- **코드 예제**: 핵심 로직을 보여줄 때 코드 블록을 사용한다. 설명 없는 코드 덩어리는 피한다.

#### 마크다운 규칙

- `<Header />` 컴포넌트를 frontmatter 바로 다음에 넣는다
- `[[toc]]`를 Header 다음에 넣는다
- `<br/>` 사용 금지 — `<br>` 사용
- `**bold**한글` 패턴 금지 — `<strong>bold</strong>한글` 사용
- GitHub 스타일 callout(`> [!TIP]`) 사용 금지 — VuePress에서 렌더링되지 않음
- VuePress 마크다운 컨테이너 활용:
  - `:::tip 제목` — 핵심 정리, 팁
  - `:::info 제목` — 참고 정보
  - `:::warning 제목` — 주의사항
  - `:::details 제목` — 접기/펼치기
- Mermaid 다이어그램 (`stateDiagram-v2`에서 `\n` 대신 `<br>` 사용)

#### 이미지

포스트에 이미지가 필요한 경우:

1. **Mermaid 다이어그램**: 흐름도, 시퀀스 다이어그램 등은 마크다운 내 Mermaid 코드 블록으로 직접 삽입
2. **외부 이미지 필요 시**: `docs/.vuepress/public/images/{slug}/` 디렉토리에 저장하고 `![alt](/images/{slug}/filename.png)` 로 참조
3. **이미지 검색/생성**: 개념도나 아키텍처 다이어그램이 필요하면 Mermaid로 최대한 표현하고, Mermaid로 불가능한 경우에만 외부 이미지를 사용

### 4단계: 핑퐁 다듬기

초안을 보여주고 사용자의 피드백을 받는다. 피드백 반영 시:

- 사용자가 특정 섹션만 언급하면 해당 섹션만 수정본을 보여준다 (전체를 다시 출력하지 않는다)
- 사용자가 "이 부분 더 자세히"라고 하면 해당 섹션을 보강한다
- 사용자가 "이건 빼"라고 하면 삭제한다
- 사용자가 내용을 직접 쓰면 그 문체를 존중하고 Claude 스타일로 고치지 않는다

핑퐁은 사용자가 만족할 때까지 반복한다.

### 5단계: 최종 확인

글이 완성되면:

1. 최종본의 frontmatter를 보여준다
2. 카테고리, 제목, 태그, description을 확인받는다

```
최종 정보:
- 카테고리: infra
- 파일명: 2026-04-13-kms.md
- 제목: "AWS KMS 키의 원리와 사용"
- 태그: aws, kms
- description: "KEK와 DEK, 봉투 암호화, 키 로테이션을 알아보자"

이대로 저장할까요?
```

### 6단계: 저장 및 배포

사용자가 승인하면:

1. 포스트 파일 저장: `docs/_posts/{category}/{date}-{slug}.md`
2. 이미지가 있으면 이미지 디렉토리도 함께 저장
3. 커밋 및 푸시:

```bash
cd /Users/hobeen/private/hobeen-kim.github.io
git add docs/_posts/{category}/{filename}
git add docs/.vuepress/public/images/{slug}/ # 이미지가 있는 경우
git commit -m "docs: {제목 요약}"
git push origin master
```

저장 완료 메시지:

```
저장했습니다. https://hobeen-kim.github.io/posts/{category}/{slug} 에서 확인할 수 있어요.
```

## 파일 템플릿

```markdown
---
title: "포스트 제목"
date: YYYY-MM-DD
tags:
  - 태그1
  - 태그2
description: "포스트 한 줄 요약"
---

<Header />

[[toc]]

도입부 — 이 글을 쓰게 된 배경

## 1. 첫 번째 섹션

본문 내용

## 2. 두 번째 섹션

본문 내용

## 정리

마무리

<Footer />
```

## 주의사항

- 포스트의 주인은 사용자다. Claude는 초안 작성과 편집을 돕는 역할이다.
- 사용자가 직접 쓴 문장이 있으면 그 톤을 기준으로 삼는다.
- 공식문서를 그대로 번역하는 식의 글은 피한다. 사용자의 경험과 관점이 담겨야 한다.
- 사용자가 "이 정도면 됐다"라고 하면 완벽하지 않아도 5단계로 넘어간다.
- 새 카테고리가 필요하면 사용자에게 확인 후 생성한다.
- 이모지를 사용하지 않는다. 본문, 다이어그램, 제목 어디에서도 이모지 금지.
- 따옴표 사용을 최소화한다. 강조가 필요하면 볼드나 마크다운 컨테이너를 활용한다.
- 외부 블로그/아티클을 적극적으로 참고하되, 본문 중간에 링크를 넣지 않는다. 글 마지막에 `## Ref.` 섹션을 두고 참고 링크를 모아서 정리한다.
- 본문 내에서 출처를 언급하지 않는다. "A의 글에 따르면", "B는 이렇게 말했다" 같은 인용 언급 없이, 내용만 자연스럽게 녹여서 쓴다.
