OAuth 2.0 + OpenID Connect

"이 앱이 당신의 구글 캘린더에 접근해도 될까요?" 모든 로그인 버튼 뒤에는 OAuth가 있다. 이 스터디는 비밀번호를 공유하던 시절의 불편함에서 출발해, OAuth 2.0이 어떻게 권한 위임 문제를 풀었는지, OpenID Connect가 왜 추가로 필요했는지, 그리고 실제 서비스에서 부딪히는 보안·운영 이슈까지 체계적으로 학습한다.

비개발자가 "로그인이 어떻게 동작하는가"를 이해하는 수준부터, 시니어 백엔드가 PKCE·RTR·BFF 패턴을 설계하는 수준까지 한 권으로 커버한다.

목차

왜 OAuth가 필요한가

  1. 비밀번호를 공유하던 시절 — 로그인의 진화와 제3자 접근의 위험
  2. 세션·쿠키 vs 토큰 — 상태 관리의 두 갈래, stateful과 stateless
  3. 인증과 인가는 어떻게 다른가 — Authentication과 Authorization의 분리
  4. OAuth는 무엇을 해결했는가 — "대신 접근" 권한 위임 모델

OAuth 2.0 핵심

  1. 4가지 역할과 용어 — Resource Owner / Client / AS / RS
  2. Authorization Code Flow — 표준 플로우와 Device Authorization Grant
  3. 나머지 Grant Type과 한계 — Implicit / ROPC / Client Credentials
  4. Access·Refresh Token 수명 주기 — 발급·검증·Introspection·Revocation

OpenID Connect

  1. OIDC는 왜 태어났나 — OAuth 인증 오남용과 표준화
  2. ID Token과 JWT 구조 — 헤더·페이로드·서명과 검증 절차
  3. Discovery · JWKS · UserInfo — 표준 스코프와 키 로테이션

보안 심화

  1. PKCE — 공개 클라이언트 코드 가로채기 방어
  2. 토큰 수명 관리 전략 — Refresh Token Rotation · DPoP · mTLS
  3. 토큰 저장·전송과 BFF 패턴 — SPA 권장 아키텍처
  4. OAuth를 노리는 공격들 — Open Redirect · State CSRF · Mix-up

실전 구현·운영

  1. Spring Security OAuth2 Client — 소셜 로그인 함정들
  2. Keycloak으로 AS 구축 — Dynamic Client Registration 포함
  3. SSO와 Federation — SAML 비교와 IdP 연동
  4. 관찰성과 감사 — 토큰 라이프사이클 로깅과 이상 탐지
  5. OAuth의 미래 — GNAP · Passkey · WebAuthn
