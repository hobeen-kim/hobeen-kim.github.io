CH9. 오류 감지·격리

학습 목표

  • CAN이 감지하는 5가지 오류의 정의와 각각이 발생하는 시점을 구분할 수 있다.
  • TEC(송신 오류 카운터)와 REC(수신 오류 카운터)의 증감 규칙을 이해한다.
  • Error Active / Error Passive / Bus-off 세 상태의 전이 조건과 행동 차이를 설명할 수 있다.
  • Fault Confinement가 왜 네트워크 전체의 건강을 지키는지, 그리고 실무에서 무엇을 로깅해야 하는지 안다.

1. CAN의 오류 감지 철학

CAN은 자가 감시(self-policing) 버스다. 중앙 관리자가 없어도 각 노드가 스스로 송수신을 관찰하다가 이상을 발견하면 Error Frame으로 알린다. 그리고 고장이 의심되는 노드는 스스로 조용해지는 구조로 설계됐다. 한 노드의 고장이 전체를 망가뜨리지 못하게 하는 이 메커니즘을 Fault Confinement라고 부른다.

핵심은 두 가지다. 하나는 오류를 어떻게 감지하느냐, 다른 하나는 감지된 오류로 누구를 어떻게 격리하느냐다. 앞쪽은 5가지 오류 종류로, 뒤쪽은 TEC/REC 카운터와 3단계 상태 전이로 풀어낸다.

CAN의 오류 처리는 이더넷이나 일반 네트워크 프로토콜과 근본적으로 다르다. 일반 네트워크는 상위 계층(TCP 등)에서 재전송으로 복구하지만, CAN은 링크 계층에서 원자적으로 오류를 감지하고 자동 재송신까지 처리한다. 응용 소프트웨어는 오류의 존재를 알 필요조차 없다. 덕분에 개발자는 "이 메시지가 도착했다면 내용은 무결하다"는 가정을 안전하게 유지할 수 있다. 이 무결성 보장이 CAN을 안전 크리티컬 시스템의 표준으로 만든 가장 큰 이유다.

2. 5가지 오류 — 무엇이 잘못됐는가

CAN 컨트롤러는 프레임 송수신 중 다음 다섯 가지를 감시한다.

Bit Error

송신 노드가 자기가 내보낸 비트와 버스에서 관측되는 비트가 다른지 확인한다. 다르면 Bit Error다. 단, Arbitration 필드와 ACK slot에서는 예외다. 중재 중에는 내가 recessive를 쐈는데 버스가 dominant로 끌려가는 것이 정상이고(내가 졌을 뿐), ACK slot에서는 내가 recessive를 쐈는데 수신자가 dominant로 덮어쓰는 것이 설계 그대로다.

Bit Error는 송신자에게 가장 직접적인 고장 신호다. 자기가 쏜 비트가 왜곡된다는 것은 트랜시버 드라이브 강도 부족, 케이블 저항 상승, 접지 부실, 버스 단락 중 하나를 의미한다. 특히 dominant를 쐈는데 버스가 recessive로 남는다면 드라이버 고장 또는 단선, recessive를 쐈는데 dominant로 보인다면 버스 단락이 가장 흔한 원인이다.

Stuff Error

CAN은 비트 스터핑 규칙에 따라 같은 극성의 비트가 5개 연속되면 반대 극성의 스터프 비트를 하나 삽입한다. 수신자는 이 규칙을 역으로 적용해 스터프 비트를 제거하는데, 5연속 동일 비트 뒤에 반대 비트가 오지 않으면 Stuff Error다. 비트 스터핑은 동기화 유지 수단이므로, 이 오류는 수신자의 비트 샘플링이 송신자 타이밍과 어긋났음을 의미하기도 한다. 클럭 드리프트, 오실레이터 정확도 부족, 비트 타이밍 설정 실수가 주된 원인이다.

Form Error

프레임에는 값이 고정된 형식 필드가 여럿 있다. CRC delimiter, ACK delimiter, EOF(End Of Frame), IFS 등은 recessive로 정해져 있다. 이 자리에 dominant가 관측되면 Form Error다. 실무에서는 CAN과 CAN FD가 섞인 버스에서 많이 본다. Classical 노드가 FD 프레임의 FDF 비트 위치에서 예상하지 못한 값을 보고 Form Error를 뱉는 것이다.

CRC Error

수신자가 프레임을 받아 자체 계산한 CRC와 송신자가 붙여 보낸 CRC 값이 다르면 CRC Error다. 데이터 필드가 전송 중 훼손됐다는 뜻이다. Classical CAN은 15비트 CRC를 쓰고, 이는 최대 6비트 연속 오류까지 100% 감지한다. CAN FD는 페이로드가 커진 만큼 17비트와 21비트 두 가지 CRC를 선택적으로 사용해 감지 범위를 확장했다.

Acknowledgment Error

송신자가 ACK slot에서 recessive를 쏘고 버스를 관측했을 때, 누군가 dominant로 덮어써야 정상이다. 끝까지 recessive로 남아 있으면 아무도 내 프레임을 정상 수신하지 못했다는 뜻이고, 송신자는 ACK Error를 기록한다.

ACK Error가 반복되는 전형적 상황은 버스에 혼자 깨어 있는 경우다. ECU 하나만 전원이 들어와 송신하는 bench-top 테스트, 또는 sleep 상태의 다른 노드들과 고립된 wake-up 초기 구간이 대표적이다. 이때 노드는 ACK를 받지 못해 계속 재전송하고 TEC가 빠르게 증가한다. 실무에서는 self-ACK 모드나 loopback 테스트 환경을 별도로 구축해 이런 오인지를 피한다.

오류 중 하나라도 감지한 노드는 즉시 Error Frame을 쏴서 해당 프레임을 무효화시킨다(자세한 프레임 구조는 CH6 참조).

다섯 오류가 독립적으로 동작하는 게 아니라 상호 보완 관계라는 점이 중요하다. CRC Error는 데이터 훼손을, Form Error는 동기 오류를, Stuff Error는 연속된 비트 깨짐을, Bit Error는 송신 경로의 전기적 실패를, ACK Error는 네트워크 고립을 잡는다. 어느 한 오류가 다른 오류를 대체하지 못하고, 다섯이 함께 작동해야 CAN이 주장하는 5×10⁻¹¹ 이하의 잔류 오류율(residual error rate)을 달성할 수 있다. 이 수치는 항공·자동차 안전 표준의 정량 근거로 쓰인다.

3. TEC / REC — 오류의 신용 점수

오류 하나하나가 중요한 게 아니라 빈도가 중요하다. 순간 노이즈로 인한 일회성 오류와, 케이블 단선·트랜시버 고장에서 오는 지속적 오류는 다른 대응을 받아야 한다. CAN은 이를 TEC(Transmit Error Counter)와 REC(Receive Error Counter)라는 두 카운터로 구분한다.

대체적인 증감 규칙은 다음과 같다.

  • 송신 중 오류 감지 → TEC +8
  • 수신 중 오류 감지 → REC +1
  • 수신 오류에 연속해서 또 감지 → REC +8
  • 송신 성공 → TEC −1
  • 수신 성공 → REC −1 (단 하한 0, 상한 127까지의 구간에서)

규칙이 비대칭인 이유는 명확하다. 실패(+8)는 성공(−1)의 8배 가중치를 가진다. 문제 노드는 성공을 8번 해야 실패 1번을 상쇄할 수 있다. 또 송신 오류가 수신 오류보다 크게 처벌되는 이유는, 버스에 실제로 신호를 주입하는 주체가 송신자이기 때문이다. 고장의 원인일 가능성이 더 높다.

세부 규칙에는 몇 가지 예외 조항도 있다. 예를 들어 Error Passive 노드가 ACK Error를 감지했을 때는 TEC를 증가시키지 않는다. Passive 노드는 이미 "문제 있는 노드"로 분류되어 있으므로, ACK를 받지 못해도 네트워크 전체가 문제인지 자기가 문제인지 불분명하기 때문이다. 또 Error Passive에서 송신에 성공하면 TEC는 127로 직접 감소한다. 성공 직후 Error Active로 바로 돌아가지 못하게 막는 이 규칙은, 경계 구간에서 상태가 깜빡이는 oscillation을 방지하기 위한 것이다.

카운터는 어디에 있는가

TEC/REC는 CAN 컨트롤러 내부 레지스터에 존재한다. MCAL 드라이버는 보통 Can_GetControllerErrorState() 같은 API로 이 값을 조회할 수 있다. 실전에서 필드 문제가 터졌을 때 제일 먼저 들여다보는 값이다.

4. 세 상태 — Error Active / Passive / Bus-off

TEC와 REC 값에 따라 노드는 세 상태 중 하나에 놓인다.

Error Active (TEC < 128 & REC < 128)

평시 상태다. 오류를 감지하면 Active Error Flag, 즉 dominant 6비트를 쏴서 버스의 모든 노드에 오류를 강제로 알린다. 이 6비트는 비트 스터핑 규칙(5연속 한계)을 어기므로 다른 노드들도 반드시 Error Frame을 인지하게 된다. 이 "모두가 보게 만드는" 성질이 Error Active의 핵심이다. 건강한 노드는 오류를 숨기지 않고 네트워크 전체에 알려, 손상된 프레임이 응용 레벨에 전달되지 않도록 한다.

Error Passive (128 ≤ TEC < 256 or REC ≥ 128)

의심스러운 상태다. 오류를 감지하더라도 Passive Error Flag, 즉 recessive 6비트만 쏜다. Recessive는 버스의 기본 상태와 같아서 다른 노드들에게 거의 보이지 않는다. 조용히 자기 불만만 표현하는 셈이다.

또 한 가지 중요한 제약이 있다. Error Passive 노드는 송신 후 다음 송신을 위해 IFS(3비트) + 8 recessive 추가 대기를 거쳐야 한다. Error Active 노드보다 항상 늦게 재송신에 참여한다는 뜻이다. 건강한 노드가 먼저 버스를 쓰도록 배려하는 장치다.

실무에서 Error Passive 진입은 경고 신호다. 일회성 진입은 괜찮지만, 운전 중 주기적으로 Passive로 빠지는 노드는 배선 열화·커넥터 접촉 불량·트랜시버 노화가 진행 중일 가능성이 크다. 진단 DTC(Diagnostic Trouble Code)에 이 상태 전이를 기록해 서비스센터에서 추적할 수 있게 한다.

Bus-off (TEC ≥ 256)

가장 심각한 상태다. 노드는 송신도 수신도 멈추고 버스에서 사실상 사라진다. 트랜시버 출력은 recessive에 고정되어 다른 노드의 통신을 방해하지 않는다. 이 상태는 REC로는 진입하지 않는다. 수신만 잘못됐을 때 노드를 완전히 분리할 이유는 없기 때문이다.

Bus-off는 "CAN 프로토콜이 스스로 포기한 상태"로 이해하면 된다. 재시도해봤자 버스를 망칠 가능성만 있으니 더는 시도하지 않겠다는 선언이다. 다만 시스템 관점에서는 기능 상실이므로, 상위 레이어는 이 노드의 신호를 "Invalid"로 처리하고 페일세이프 로직으로 전환해야 한다. 예를 들어 엔진 ECU가 Bus-off로 빠지면 변속기 ECU는 이전 엔진 토크 추정값을 홀드하거나, 중립 근처의 안전값으로 대체한다.

5. Bus-off 복구 — 스스로 돌아오는 법

Bus-off에 빠진 노드는 그대로 죽지 않는다. ISO 11898-1은 11 연속 recessive 비트를 128회 관측하면 Error Active로 자동 복귀하도록 정의한다. 11 recessive는 IFS(3) + 후속 버스 idle 8비트에 해당하는 길이로, 버스가 충분히 조용할 때만 채워진다. 128번을 카운트하려면 버스에 최소 수십 ms 이상의 정상 트래픽이 흘러야 한다.

다만 제조사는 이 자동 복구를 비활성화하고, ECU 관리 로직에서 명시적으로 재초기화하도록 설계하는 경우가 많다. Bus-off가 반복되는 상황은 하드웨어 문제일 가능성이 높으므로, 무작정 되돌리면 다른 노드들에게 반복 피해를 준다. AUTOSAR CanSM(Can State Manager)은 이 재초기화와 복구 지연을 상태 기계로 관리한다.

대표적인 복구 정책은 "빠른 첫 시도, 느린 재시도"다. 첫 Bus-off는 100ms 안에 재초기화해 일시적 노이즈였는지 확인하고, 두 번째는 1초, 세 번째부터는 10초 이상 간격으로 늘린다. 이후 일정 횟수(보통 3~5회) 반복되면 영구 고장으로 판단해 DTC를 기록하고 관련 기능을 비활성화한다. 이 정책은 운전 중 계속 Bus-off/복구를 반복하며 브레이크 신호가 흔들리는 상황을 막아준다.

Bus-off가 뜨면 고장이다

실험실 환경에서는 일회성 노이즈로도 Bus-off가 발생할 수 있지만, 양산 차량에서 Bus-off는 거의 항상 하드웨어 결함이다. 종단 저항, 커넥터 산화, 트랜시버 ESD 손상, 전원 리플이 원인일 확률이 높다. 카운터 로그와 함께 물리계층부터 점검해야 한다.

7. Partial Network 환경의 오류 처리

Selective Wake-up 트랜시버가 탑재된 Partial Network 환경에서는 잠든 노드가 오류 프레임에 반응하지 않는다. 트랜시버의 WUP(Wake-Up Pattern) 검출 로직은 정해진 프레임 ID와 바이트 패턴에만 반응하도록 설계돼 있어서, 임의의 dominant 6비트로는 깨어나지 않는다.

반대로 Error Passive 노드가 연달아 passive flag를 쏘더라도 깊은 잠에 든 노드들은 영향을 받지 않는다. 이는 저전력 동작에는 유리하지만, 일부 노드만 깨어 오류가 오가는 상황에서는 깨어난 노드들끼리만의 소규모 오류 도미노가 발생할 수 있다. 진단 시 "지금 몇 개 노드가 깨어 있는가"를 반드시 확인해야 한다.

8. 실전 디버깅 — 무엇을 보고 뭘 해야 하나

필드에서 간헐 통신 장애를 만나면 다음 순서로 접근한다.

  1. TEC/REC 주기 로깅 — 드라이버에서 100ms 주기로 읽어 CAN trace에 남긴다. 값이 천천히 오른다면 간헐 노이즈, 계단식으로 튄다면 특정 이벤트에 동기화된 고장이다.
  2. 상태 전이 기록 — Error Active→Passive 또는 Passive→Bus-off 전이 순간의 타임스탬프를 확보한다. 어떤 메시지 이후였는지, 어떤 시동·조작 이벤트와 겹치는지 매핑한다.
  3. 오류 종류 통계 — CAN 컨트롤러는 마지막 오류 종류를 레지스터에 남긴다. Bit Error가 압도적이면 자기 신호가 왜곡되는 것이고, CRC·Stuff Error가 많으면 외부 노이즈나 수신 타이밍 문제다.
  4. 물리계층 확인 — 종단 저항을 멀티미터로 확인(60Ω이 정상), 오실로스코프로 파형의 rising/falling edge 관찰, 버스 양단 전위 확인.

9. 흔한 함정 — 전 노드 동시 Passive

가끔 전체 노드가 동시에 Error Passive로 가는 괴이한 현상을 만난다. 원인은 크게 두 가지다.

  • Dominant 막힘: 종단 단락이나 외부 접지 쇼트로 버스가 dominant에 고정되면, 모든 노드가 자기 recessive를 쏴도 버스가 dominant로 남아 Bit Error가 쏟아진다. TEC는 아니지만 REC가 동시에 오른다.
  • Recessive 약화: bias 회로 불량이나 종단 저항 분리로 recessive 전압이 충분히 복원되지 않으면, 모든 수신자가 CRC/Form Error를 본다. 한 노드가 아니라 버스 자체가 고장이다.

개별 노드를 의심하기 전에 "전체가 동시에 나빠지는가"를 봐야 한다. 그렇다면 용의자는 케이블·종단·공통 전원이다.

또 한 가지 자주 놓치는 함정은 에러 프레임이 trace에 안 찍히는 경우다. 저가형 USB-CAN 인터페이스 중에는 error frame을 드롭해버리고 정상 프레임만 상위에 전달하는 제품이 있다. 이런 환경에서는 TEC/REC가 튀는데 trace는 깨끗해 보여 원인 추적이 어려워진다. 필드 진단용 인터페이스는 error frame passthrough와 상태 레지스터 로깅을 지원하는 제품을 선택해야 한다.

다음 챕터

Fault Confinement로 죽은 노드를 분리하는 것까지 봤다. 이제 살아 있는 노드들끼리의 경쟁을 다뤄야 한다. CH10에서는 버스 부하와 응답시간을 수식과 사례로 풀어 실시간 시스템에서 CAN을 어떻게 설계할지 다룬다.

핵심 정리

  • CAN은 Bit·Stuff·Form·CRC·ACK 5가지 오류를 감지하며, 감지 주체와 시점이 오류별로 다르다.
  • TEC/REC는 +8(실패) / −1(성공)의 비대칭 규칙으로 쌓여, 문제 노드가 자연스럽게 격리되게 만든다.
  • Error Active는 dominant flag로 강하게, Error Passive는 recessive flag로 조용히, Bus-off는 완전히 침묵한다.
  • Bus-off 복귀는 128×11 recessive 관측 후 자동 또는 수동 재초기화로 이루어지며, 양산에서 반복되면 하드웨어 고장을 의심해야 한다.
  • 실전 디버깅은 TEC/REC 로깅 → 상태 전이 시점 → 오류 종류 통계 → 물리계층 순으로 접근한다.
